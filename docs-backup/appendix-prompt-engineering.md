# 📎 附录：AI 辅助学习指南 & Prompt 工程手册

## 为什么需要这份附录？

在本教程的每一章，你都在使用 AI 辅助学习。但"会用 AI"和"用好 AI"之间，隔着一整套 **Prompt Engineering（提示词工程）** 的技能。

这份附录将：
1. 总结 Prompt Engineering 的核心框架
2. 提供一个 Prompt 模板库，覆盖学习和开发的各种场景
3. 给出在各章节中使用 AI 辅助的具体策略

## 第 1 部分：Prompt Engineering 核心框架

### 黄金五要素（TCOF 框架）

任何优秀的 Prompt 都应该包含以下五个要素：

```
T - Task（任务）: 明确告诉 AI 要做什么
C - Context（上下文）: 提供背景信息和约束条件
O - Output（输出）: 指定输出的格式和风格
F - Format（格式）: 给定期望的格式结构

+ Role（角色）: 给 AI 分配一个角色/身份
```

**一个对比：**

```diff
- ❌ "解释什么是 RAG"
+ ✅ 
角色：你是一个 AI 技术导师，面向零基础的初学者
任务：用不超过 200 字解释什么是 RAG
上下文：读者已经了解基本的 LLM API 调用，但不知道 RAG
输出：先用一句话类比，再给一个技术定义，最后用 3 个要点说明为什么需要它
格式：使用 Markdown 段落格式，不要用列表
```

### 六大核心技巧

#### 1. 角色指定（Role Prompting）

给 AI 分配一个身份，让它以这个身份的知识背景和表达方式来回答。

```python
# 通用
"帮我写一段 Python 代码"
# → 得到泛泛的、不一定符合你需求的代码

# 角色化
"你是一个有 10 年 Python 经验的资深工程师，注重代码可读性和类型安全。帮我写一段调用 OpenAI API 的代码。"
# → 得到更专业的、带类型注解和错误处理的代码
```

**什么时候用：** 几乎所有场景。角色指定几乎总是能提升输出质量。

#### 2. 少样本示例（Few-Shot）

提供一个或多个输入-输出示例，让 AI 模仿你的模式。

```
任务：将用户问题分类为"天气"/"计算"/"搜索"
示例 1：
输入：北京今天多少度？
输出：天气

示例 2：
输入：2500 * 12 等于多少？
输出：计算

示例 3：
输入：帮我查一下量子计算的最新进展
输出：搜索

现在分类：
输入：深圳明天会下雨吗？
输出：
```

**什么时候用：** 需要特定格式、风格、或者分类标准不太容易用语言描述的场景。

#### 3. 思维链（Chain-of-Thought）

让 AI 展示推理过程，而不是直接给答案。

```diff
- ❌ "计算 25 * 4 + 100 / 2"
+ ✅ "计算 25 * 4 + 100 / 2，请分步展示计算过程"
```

**什么时候用：** 复杂推理、数学计算、多步骤分析、需要可解释性的场景。

#### 4. 结构化输出（Structured Output）

明确指定输出的格式，让你的代码可以可靠地解析结果。

```
请按以下 JSON 格式返回：
{
  "summary": "一句话总结",
  "key_points": ["要点1", "要点2", "要点3"],
  "sentiment": "positive/neutral/negative"
}
```

**什么时候用：** 需要代码解析 AI 输出的场景（绝大部分 Agent 场景）。

#### 5. 任务分解（Task Decomposition）

把复杂任务拆成多个简单的 Prompt。

```diff
- ❌ "帮我分析这份数据，写报告，然后发邮件"
+ ✅ "第一步：帮我总结这份数据的关键指标"
+     "第二步：基于总结写一份报告"
+     "第三步：把报告作为邮件正文发送"
```

**什么时候用：** 任务步骤之间耦合度低，或者单次 Prompt 无法完成。

#### 6. ReAct（Reasoning + Acting）

让 AI 在推理和行动之间交替。这是 Agent 的核心模式。

```
用户问题：北京的天气怎么样？

思考：用户想知道北京的天气，我需要查询天气工具。
行动：调用 get_weather(city="北京")
观察：返回"晴，25°C"
思考：天气信息已经够回答了。
回答：北京今天晴，25°C。
```

**什么时候用：** 构建 Agent 系统时。

## 第 2 部分：各章节专属 Prompt 模板库

### 第 1 章：学 Python

```
角色：你是一个擅长类比教学的编程导师
任务：用生活类比解释 [Python 概念：如字典、列表]
受众：零编程基础的学习者
要求：必须使用一个生活场景的比喻，先解释概念，再给代码示例
输出格式：概念类比 → 代码示例 → 逐行解释
```

```
角色：你是一个严格的 Python 代码审查员
任务：审查以下代码，指出风格问题和优化空间
我的代码：[贴代码]
审查重点：命名规范、异常处理、代码可读性
输出格式：按严重程度排序（严重/建议/小问题），每条附带修改建议
```

### 第 2 章：LLM API 调用

```
角色：你是一个 API 调试专家
任务：帮我解决 API 调用错误
上下文：我正在调用 [模型名] 的 API，框架是 [Python/JavaScript]
错误信息：[贴错误信息]
我已经尝试过：[列出你尝试过的方案]
输出格式：最可能原因 → 验证方法 → 解决方案 → 预防措施
```

```
角色：你是一个 AI 应用开发导师
任务：帮我理解 [参数名，如 temperature] 的作用
前提：我已经会基础 API 调用
要求：用可实验的方式解释，即"你把 temperature 设成 0 和 1 各试一次，观察区别"
```

### 第 3 章：RAG

```
角色：你是一个 RAG 系统架构师
任务：帮我诊断 RAG 检索效果问题
我的场景：[描述你的文档类型和用户问题]
当前效果：[描述检索结果 - 是否相关]
我的配置：[切块大小、Embedding 模型、Top-K 值]
输出格式：可能原因排行 → 每个原因的验证方法 → 改进方案
```

```
角色：你是一个 RAG 开发工程师
任务：帮我改写以下代码，换成 [ChromaDB/Pinecone/Weaviate] 作为向量库
当前代码：[贴你的代码]
要求：保持功能不变，只替换向量库
```

### 第 4 章：Agent

```
角色：你是一个 Agent 系统开发专家
任务：帮我设计工具集
场景：[描述你的应用场景]
要求：列出每个工具的名称、描述、参数（JSON Schema 格式）
输出格式：工具列表（按优先级排序），每个工具附上选择理由和实现要点
```

### 第 5-6 章：Memory & Multi-Agent

```
角色：你是一个 AI 系统架构师
任务：评审我的 [Memory / Multi-Agent] 设计方案
我的方案：[描述你的设计]
评审维度：可扩展性、错误处理、成本、复杂度
输出格式：优势列表 → 风险列表 → 改进建议
```

### 第 7 章：面试准备

```
角色：你是一个资深 AI 应用开发面试官
任务：对我进行模拟面试
我的背景：[你的背景：如 1 年 Python 经验，做过 1 个 RAG 项目]
面试方向：AI 应用开发 / Agent 开发
要求：先问 3 个基础关，再问 2 个深入题，最后给综合反馈
```

## 第 3 部分：常见错误与改进

### 错误 1：Prompt 太模糊

```diff
- ❌ "写一段代码"
+ ✅ "用 Python 写一个函数，调用 OpenAI API 实现多轮对话，包含对话历史管理"
```

### 错误 2：不给角色

```diff
- ❌ "解释什么是 Agent"
+ ✅ "你是一个 AI 技术布道师，用通俗易懂的方式解释什么是 Agent。面向非技术人员。"
```

### 错误 3：不指定输出格式

```diff
- ❌ "对比 RAG 和微调的优缺点"
+ ✅ "用表格对比 RAG 和微调在以下维度：适用场景、成本、知识更新速度、准确性"
```

### 错误 4：一次性要求太多

```diff
- ❌ "帮我做一个完整的电商客服 Agent"
+ ✅ 分步骤：
+   "第一步：设计客服 Agent 的工具集"
+   "第二步：实现查询订单工具"
+   "第三步：实现退货处理流程"
```

### 错误 5：不提供上下文

```diff
- ❌ "代码报错了"
+ ✅ "我在运行以下代码时遇到了错误：[代码]。运行环境：Python 3.10, openai 1.0。错误信息：[贴错误]。我在 Windows 上运行。"
```

## 第 4 部分：AI 辅助学习策略

### 学习四步法

```
1. 预习：让 AI 用 100 字概括今天要学的概念
   ↓
2. 阅读教程：理解核心内容
   ↓
3. 实践：敲代码，跑通为止
   ↓
4. 巩固：让 AI 出题测验，或者让 AI 审查你的代码
```

### 当卡住时

```
你是否已经：
1. 把错误信息复制给了 AI？              → 去做
2. 把自己的代码和期望输出给了 AI？       → 去做
3. 告诉 AI 你试过了什么方案？           → 去做
4. 把相关章节的内容告诉了 AI？          → 去做

如果以上都做了还是卡住 → 跳过，先往后学，回头再看
```

### 不要这样做

```diff
- ❌ 把 AI 当成"搜索引擎"来用
- ❌ 遇到问题不思考直接问 AI
- ❌ 复制 AI 的代码但不理解就运行
- ❌ 从不自己敲代码
- ❌ 收藏大量资源但从不看
```

## 第 5 部分：Prompt 快速参考卡

| 场景 | Prompt 模板 |
|------|------------|
| 概念解释 | "你是一个导师，用 [类比] 解释 [概念]，面向零基础" |

| 出练习题 | "角色：导师。主题：[主题]。难度：从易到难。数量：5 题。附带答案" |

## 最后的提醒

**Prompt Engineering 不是一个需要"学完"的技能，它是一个在实践中不断优化的过程。**

好的 Prompt 不是一次写成的，而是**迭代出来的**：

```
写 Prompt → 看结果 → 发现问题 → 修改 Prompt → 再看结果 → ...
```

你的第一个 Prompt 可能不完美，没关系。**每次用 AI 都是练习 Prompt Engineering 的机会。**

记住加一的座右铭：

> **"先跑通，再优化。先做成，再做好。"**
