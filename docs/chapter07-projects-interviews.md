## 三个必做项目

### 项目 1：知识库问答助手（RAG 项目）

**难度：** ⭐⭐
**与章节对应：** 第 3 章

**项目描述：**
做一个能上传文档、基于文档内容回答问题的系统。支持 PDF/TXT 格式，能展示检索到的原文片段。

**技术要求：**
- 文档解析与切块
- Embedding + 向量库（FAISS / ChromaDB）
- 检索 + 生成（RAG 完整链路）
- 前端展示（Streamlit / Gradio）

**加分项：**
- 支持多种文档格式（PDF、Word、Markdown）
- 中文优化（中文分词、更好的 Embedding 模型）
- 显示引用来源（展示来自哪个文档的哪个段落）

**简历描述模板：**
> 基于 RAG 架构开发智能知识库问答系统，支持多格式文档上传、语义检索和上下文增强生成，检索准确率达 XX%，回答采纳率 XX%。

### 项目 2：智能助手 Agent（工具调用项目）

**难度：** ⭐⭐⭐
**与章节对应：** 第 4 章 + 第 5 章

**项目描述：**
做一个能调用多种工具的 AI 助手。至少支持：搜索、计算、天气查询、备忘录、待办事项管理。

**技术要求：**
- Function Calling / Tool Calling
- 多工具注册与管理
- 对话记忆（短期 + 长期）
- 错误处理与重试机制

**加分项：**
- 支持 MCP 协议（工具标准化接入）
- 工具调用的中间步骤可视化展示
- 用户自定义工具（让用户添加自己的 API）

**简历描述模板：**
> 设计并实现多工具 AI Agent 系统，支持 X 种工具的动态注册与调用，引入记忆管理机制使多轮对话准确率提升 XX%，日均处理请求 XX 条。

### 项目 3：接近真实业务场景（综合项目）

**难度：** ⭐⭐⭐⭐
**与章节对应：** 全部

**选择以下任一方向：**

#### 方向 A：运营提效助手

**场景：** 运营团队需要一个助手，能查询数据、生成报表、发送邮件。

**功能：**
- 连接到数据库/API 查询实时数据
- 根据模板生成报表
- 发送邮件给指定人员
- 定时任务调度

#### 方向 B：数据分析助手

**场景：** 用户上传 Excel/CSV 数据，用自然语言提问数据分析问题。

**功能：**
- 数据上传与预览
- 自然语言转 SQL 或 Pandas 代码
- 图表生成
- 数据洞察报告

#### 方向 C：个人知识管理助手

**场景：** 收集和整理零散信息，构建个人知识库。

**功能：**
- 网页内容抓取与摘要
- 标签分类与自动归档
- 基于知识库的问答
- 定期回顾系统

**简历描述模板：**
> 独立设计与开发 [场景] AI 助手，整合 RAG + Agent + 工具调用技术栈，支撑 [业务指标] 提效 XX%，获得 [团队/客户] 正面反馈。

## 面试高频问题 TOP 20

### 基础概念类

**Q1：什么是 RAG？为什么需要 RAG？**
> **参考：** RAG = Retrieval-Augmented Generation。在模型生成前先检索相关知识库，让回答建立在事实基础上，减少幻觉。适合需要引用私有知识或最新信息的场景。

**Q2：RAG 和微调有什么区别？**
> **参考：** RAG 是在推理时检索外部知识，不改变模型参数；微调是更新模型参数。RAG 适合动态知识、冷启动快；微调适合改变模型行为风格。

**Q3：什么是 Agent？Agent 和 Chatbot 有什么区别？**
> **参考：** Agent = Chatbot + Tools + Loop。Chatbot 只能对话，Agent 能感知环境、做出决策、执行行动。

**Q4：什么是 Function Calling？原理是什么？**
> **参考：** Function Calling 是 LLM 输出结构化工具调用请求的能力。LLM 根据工具描述返回 JSON 格式的"工具名+参数"，开发者执行并返回结果。

**Q5：什么是 MCP？解决什么问题？**
> **参考：** Model Context Protocol，AI 工具接入的开放标准。解决工具接入碎片化问题，提供统一的工具发现和调用接口。

**Q6：什么是 Embedding？**
> **参考：** 将文字转换成固定维度的向量，让语义相近的文字在向量空间中距离更近。用于语义搜索、聚类等。

**Q7：什么是 ReAct 模式？**
> **参考：** Reasoning + Acting 的缩写。Agent 在推理和行动之间交替：思考需要什么 → 执行行动 → 观察结果 → 继续思考。

**Q8：什么是幻觉（Hallucination）？如何缓解？**
> **参考：** 模型生成看似合理但实际错误的内容。缓解方式：RAG（提供事实依据）、更好的 Prompt（限制角色）、降低 Temperature。

### 实践类

**Q9：描述你做过的一个 Agent 项目，架构是怎样的？**
> **准备：** 选一个你实际做过的项目，描述：场景 → 架构 → 技术选型 → 核心实现 → 遇到的问题 → 如何解决。

**Q10：如何处理 Agent 的异常情况？**
> **参考：** 超时重试、工具调用的 try/catch、最大步数限制、降级策略（工具不可用时给出友好提示）。

**Q11：如何设计一个工具（Tool）的接口？**
> **参考：** 单个职责、清晰的名称和描述、类型注解、合理的参数设计、完善的错误处理。

**Q12：如果 Agent 陷入了死循环怎么办？**
> **参考：** 设置最大迭代步数、检测重复行为（连续相同输出时中断）、加入"无工具可调时直接回答"的兜底逻辑。

**Q13：长对话中上下文窗口满了怎么办？**
> **参考：** 滑动窗口（保留最近 N 轮）、总结压缩（Summarization）、按相关性丢弃（基于 Embedding 选择重要内容保留）。

**Q14：RAG 检索效果不好，怎么排查？**
> **参考：** 检查切块策略（是否切断语义）、检查 Embedding 模型（是否适合你的语言/领域）、检查查询重写（用户问题是否需要改写后再检索）。

### 设计类

**Q15：设计一个客服 Agent，你会怎么做？**
> **准备：** 从需求分析 → 架构设计 → 工具设计 → 知识库设计 → 记忆策略 → 异常处理，完整阐述。

**Q16：如何评估一个 Agent 系统的好坏？**
> **参考：** 任务完成率、平均轮数、工具调用准确率、用户满意度、端到端延迟、Token 消耗。

**Q17：多 Agent 和单 Agent 怎么选？**
> **参考：** 任务是否需要不同角色/专业知识？是否需要并行处理？单 Agent 上下文是否经常不够用？满足 2 个以上考虑多 Agent。

**Q18：如何保证 Agent 的安全性？**
> **参考：** Prompt 注入防护、工具调用的权限控制、敏感信息过滤、输出内容审核、人工确认机制（高风险操作）。

**Q19：如果让你选一个 Agent 框架，你会选哪个？为什么？**
> **参考：** LangChain（生态丰富）、Semantic Kernel（微软生态）、CrewAI（多 Agent 友好）、AutoGen（微软多 Agent）。说出你的理由。

**Q20：未来 1-2 年 AI 应用开发的方向是什么？**
> **参考：** Agent 工程化（MCP 等标准）、多模态 Agent（文本+图像+语音）、垂直领域深度 Agent、端侧 Agent。

|------|------|------|

| **R**esult | 结果 | 查询时间从 3 小时降到 10 分钟 |
